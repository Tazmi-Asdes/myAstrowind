## 搭建服务器的原因

我需要拥有一台服务器，用来托管个人博客、项目作品集，以及学习和熟悉运维相关操作。我决定用一台闲置的 Windows 主机，安装 Ubuntu Server 操作系统，开始实践。

## 第一阶段：系统安装与基础网络配置

### 1. 制作启动盘与系统安装

用 Rufus 制作 Ubuntu Server 24.04 LTS 启动盘。安装过程中遇到了显卡驱动兼容性问题，GTX 650 在 Ubuntu 下显示异常，通过在启动参数中添加 `nomodeset` 临时解决了问题，成功进入安装界面。系统安装时选择整盘自动分区并启用 LVM，简化了磁盘管理。

### 2. 网络配置与外网访问

系统安装后，我固定了服务器的内网IP，确保每次重启后地址不变。随后配置了 DDNS 服务，使用 `ddns-go` 将域名与宽带的动态公网 IP 绑定。因为宽带支持 IPv6，我直接通过 IPv6 地址实现外网访问，无需配置端口转发。

## 第二阶段：搭建管理面板与博客系统

### 1. 部署 1Panel 管理面板

在宝塔和1panel这两种管理面板选择中，我选择了1panel。因为我认为1panel轻量化程度更高且提供了一个图形化界面，可以方便地管理 Docker 容器、网站和服务器状态。
安装命令如下：

```bash
curl -sSL https://resource.fit2cloud.com/1panel/package/quick_start.sh -o quick_start.sh && sudo bash quick_start.sh
```

安装完成后，通过面板部署各种服务，降低了后续的运维复杂度。

### 2. 从 Halo 迁移至 Astro

最初我通过 1Panel 安装了 Halo 博客系统，并将其绑定到子域名上。Halo 的图形化后台和插件生态让初期使用非常顺手。但后来我想有更高的定制自由度和更好的性能表现，于是决定迁移到 Astro 静态站点生成器。

### 3. 部署 AstroWind 作品集站点

我在本地克隆了 AstroWind 项目，这是一个基于 Astro 的作品集模板，支持博客、作品展示和深色模式。

## 第三阶段：运维实践与自动化

### 1. 定时清理 Docker 日志

Docker 容器在运行过程中会持续产生日志，如果不加清理，日志文件可能占满硬盘空间。我通过 cron 配置了一条定时任务，每周日凌晨自动清空所有容器的日志文件：

```bash
0 3 * * 0 sudo find /var/lib/docker/containers/ -name "*.log" -exec truncate -s 0 {} \;
```

### 2. 防火墙策略

我启用了 ufw 防火墙，仅开放了 SSH、1Panel 面板和网站服务所需的端口。对于 IPv6 入站流量，由于路由器的 IPv6 防火墙默认拦截外部请求，我调整了入站策略，同时通过 ufw 控制服务器开放的端口。

### 3. 备份策略

通过 1Panel 的计划任务，将网站目录和数据库定期备份到阿里云 OSS 对象存储中，并设置保留最近 3 份备份，避免占用过多存储空间。

## 第四阶段：遇到的问题与解决思路

在整个搭建过程中，我遇到了几个关键问题，整理如下：

第一个问题是显卡驱动导致 Ubuntu Server 安装界面黑屏。这是因为 GTX 650 显卡与 Ubuntu 默认的开源驱动兼容性不佳。我通过在 GRUB 启动参数中添加 nomodeset 成功进入安装环境，后续也一直使用系统自带的 nouveau 驱动，没有额外安装闭源驱动，避免了内核升级带来的兼容性风险。

第三个问题是外网访问网站时连接超时。经过排查，发现是路由器的 IPv6 防火墙默认拦截了所有入站请求。将防火墙级别调整后，外网通过 IPv6 直接访问服务器恢复正常，同时服务器的 ufw 依然对开放的端口进行细粒度控制，安全性没有降低。

## 目前情况

从零开始搭建了一台可以公网访问的服务器，经历了操作系统安装、网络配置、Docker 容器化部署、静态站点生成、DDNS 动态域名解析、防火墙策略配置以及自动化运维脚本编写等环节。

目前运行稳定，后续计划在此基础上接入监控系统（如 Prometheus + Grafana）。
